\chapter{Implementation}\label{sec:implementation} % approx. 9 pages
The following chapter outlines the implementation of the core components from the proposed \gls{hrl} framework, comprising the high-level \gls{rdc} agent, the low-level \gls{uic} agents, and the Demand Forecasting Module, as specified in the previous chapter.

All computational tasks, including data preparation, training of the agents/models, hyperparameter optimization, and evaluation, were performed on a machine provided by the KEIM application centre, part of the University of Esslingen and the Fraunhofer IAO.
This machine is powered by an Intel Core i7-6850K CPU (six cores, each at 3.60 GHz), four NVIDIA GeForce GTX 1080 Ti graphics cards (each with 12 GB of VRAM), and 64 GB of RAM.

\section{Regional Distribution Coordinator (RDC)}\label{sec:implementation_rdc}
The \glsfirst{rdc} defines the \gls{rl} agent of the first level of the proposed \gls{hrl} framework (see \autoref{sec:design-rl_framework}).
The central component of the agent is the \texttt{RegionalDistributionCoordinator} class, which defines the agent itself.
The agent utilizes an \gls{mh-dqn}, defined by the class \texttt{MultiHeadDQN}, to choose rebalancing quotas, thereby tackling inter-community imbalances in the system through manual rebalancing by service workers.
To ensure efficient learning and effective use of training data, the agent employs a \gls{per} \cite{schaul_prioritized_2016} buffer, which is implemented by the \texttt{ReplayBuffer} class.
The experiences stored in this buffer are gained by interacting with the \texttt{EScooterRDCEnv}, which defines a custom environment based on the OpenAI Gym interface \cite{brockman_openai_2016}.

The configurable hyperparameters that influence the agent's ability to learn an efficient rebalancing strategy are defined in \autoref{tab:rdc_hyperparameters_description}.

\subsection{Multi-Head Deep Q-Network Architecture}
The \texttt{MultiHeadDQN} class defines the architecture of the underlying network of the \gls{rdc} agent.
This class inherits from \texttt{torch.nn.Module}.
The architecture of this \gls{mh-dqn} is defined as follows:

\begin{itemize}
    \item \textbf{Input Layer}:
        The input state tensor is passed to a \texttt{torch.nn.LayerNorm} layer to standardize the feature values, which stabilizes the learning process, by ensuring all input features have the same scale \cite{ba_layer_2016}.
        This input normalization is a common practice in deep learning models.
    \item \textbf{Shared Layers}:
        The normalized state tensor then gets processed by two fully connected layers with a size of \texttt{RDC\_hidden\_size}.
        Each of these fully connected shared layers (\texttt{torch.nn.Linear}) are followed by a non-linear \texttt{torch.nn.ReLU} activation function.
        By processing the combined state of all communities, these layers learn to identify system-wide patterns and spatio-temporal interdependencies.
        This network of shared layers outputs a condensed, high-level feature representation of the entire system, which is then passed to the specialized heads.
    \item \textbf{Head-Specific Layer}:
        The feature tensor outputted from the shared layers is then processed in $N_C$ heads, one head for each community.
        Each of these heads consists of two fully connected linear layers (\texttt{torch.nn.Linear}) and a \texttt{torch.nn.ReLU} activation function in between.
        The second of those layers outputs the Q-values for each discrete action per community.
        This branching architecture allows the agent to learn specific value functions for each of the community's action spaces, which is more efficient than a single monolithic approach with a combined action space \cite{li_personalized_2024}.
\end{itemize}

This network is embedded within a custom simulation environment, which is described next.

\subsection{Environment}
The \texttt{EScooterRDCEnv} class implements the simulation environment used for training the \gls{rdc} agent.
It follows the standard OpenAI Gym interface \cite{brockman_openai_2016}.
The \texttt{step()} function defines the core logic of the simulation, which is detailed in the following:

\begin{enumerate}
    \item \textbf{Action execution:}
        The \texttt{step()} method accepts an action vector $\mathcal{A}_t^{RDC}$ as input.
        This vector contains the chosen rebalancing quotas from the \gls{rdc} agent.
        To ensure vehicle conservation, as defined in the rebalancing problem (\autoref{sec:design-problem_definition}), these quotas are scaled so that the sum of all quotas is zero, while maintaining the intended flow.
    \item \textbf{Rebalancing Simulation:}
        After scaling the rebalancing quotas, the rebalancing is performed with these scaled quotas.
        For communities with a positive quota, vehicles are added to the zones with the lowest vehicle counts first.
        Conversely, for communities receiving a negative quota, vehicles are removed from zones with the highest vehicle counts first.
        This simplified approach for allocating vehicles within a community assumes that operators prioritize balancing the most extreme zone-level imbalances first.
    \item \textbf{Demand Simulation:}
        After the vehicle counts are updated based on the rebalancing actions chosen by the agent, the user demand is simulated for the entire step (e.g., 60 minutes).
        Crucially, the demand simulated within the environment is drawn from raw historical data, not the forecasts provided to the agent.
        This introduces a natural degree of uncertainty and partial observability, testing the agent's robustness against the inevitable errors of the Demand Forecasting Module.
    \item \textbf{Observation and Reward Calculation:}
        After both rebalancing and demand are simulated, the reward for the timestep is calculated.
        As detailed in \autoref{sec:design-rl_framework-rdc}, the reward consists of the SDR, a penalty for the number of vehicles rebalanced, and the Gini coefficient to reward equally distributed vehicles.
        Then, a new state observation $\mathcal{S}_{t+1}^{RDC}$ is created based on the updated vehicle counts and newly forecasted demand.
        The method then returns the complete experience tuple \texttt{(state, action, reward, next\_state, done)}.
\end{enumerate}

With the agent's architecture and the environment defined, the training procedure is used to enable the agent to learn an effective rebalancing policy, which is described in the following subsection.

\subsection{Training procedure}
The training procedure of the \gls{rdc} agent is defined by the \texttt{train} method of the \texttt{RegionalDistributionCoordinator} class.
The following section describes the step-by-step implementation of this training procedure:

The initial step of the training process starts with sampling experiences from the \texttt{ReplayBuffer}.
As detailed in \autoref{sec:design-rl_framework-rdc}, this replay buffer implements \gls{per} \cite{schaul_prioritized_2016}.
The sampling is biased towards experiences with high \gls{td-error}.
This bias is controlled by the prioritization exponent \texttt{RDC\_per\_alpha}.
\gls{is} weights are used to counteract this bias.
These weights are computed using the \texttt{RDC\_per\_beta} for controlling the impact of these.
The \texttt{RDC\_per\_beta} is annealed over the course of \texttt{RDC\_per\_beta\_frames} training steps to a final value of 1.0.

After sampling experiences from the replay buffer, the Q-values are estimated for each head of the \gls{mh-dqn}.
The policy network calculates the current Q-values based on the actions $a_c^{RDC}(t)$ taken in the observed state $\mathcal{S}_t^{RDC}$.
The target Q-value $Y_t$ is then computed using the Double DQN update rule \cite{hasselt_deep_2016}:
\begin{equation}\label{eq:ddqn-target}
    Y_t = \mathcal{R}_{t+1} + \gamma Q(\mathcal{S}_{t+1},\arg\max_{a} Q(\mathcal{S}_{t+1},a;\theta_t);\theta_t^-)
\end{equation}

The loss is calculated using the Huber loss, which has been selected due to its robustness \cite{huber_robust_1964}.
The loss is calculated for each head separately, averaged for each sample, and weighted by the \gls{is}-weights.
The final loss is the average of these weighted per-sample losses.

The Adam optimizer (\texttt{torch.optim.Adam}) updates the gradients \cite{kingma_adam_2017}.
Before the optimizer steps, the gradients are clipped to a maximum value of 10 to stabilize the training process \cite{mnih_human-level_2015}.
A learning rate scheduler updates the learning rate linearly every \texttt{RDC\_lr\_step\_size} by \texttt{RDC\_lr\_gamma}.

Finally, the agent's parameters are updated accordingly.
Using soft updates with the factor \texttt{RDC\_tau}, based on the current target network ($\theta_t^-$) and the current policy network ($\theta_t$), new values for the target network $\theta_{t+1}^-$ are calculated as follows \cite{hasselt_deep_2016}:
\begin{equation}
    \theta_{t+1}^- = \tau \theta_{t} + (1-\tau)\theta_t^-
\end{equation}
The \texttt{epsilon} value of the $\epsilon$-greedy strategy is annealed using the following formula, where $n_{step}$ is the current step count and $\delta_{\epsilon}$ the epsilon decay factor \texttt{RDC\_epsilon\_decay}: 
\begin{equation}
    \epsilon = \max(\epsilon_{min}, \epsilon_{start} \cdot (\delta_{\epsilon}^{n_{step}}))
\end{equation}
Lastly, the priorities of the \gls{per} are updated using the newly calculated \glspl{td-error} by the \texttt{update\_priorities} function from the \texttt{ReplayBuffer}.

The plot displays the episodic reward, which shows a clear upward trend before converging, indicating that the agent successfully learned a policy to maximize its objective function.

\subsection{Environment}
The \texttt{EScooterUICEnv} class implements the simulation environment used for training the \gls{uic} agents.
It follows the standard OpenAI Gym interface \cite{brockman_openai_2016}.
The \texttt{step()} function defines the core logic of the simulation, which is detailed in the following:

\begin{enumerate}
    \item \textbf{Action execution:}
        The \texttt{step()} function gets invoked with the taken action from the \gls{uic} as its single parameter.
        This action is a vector containing the incentive levels for each zone corresponding to the community.
    \item \textbf{Rebalancing Simulation:}
        The rebalancing actions are simulated first.
        Therefore, the intended drop-offs from users are taken into account, and for each drop-off event, the simulation determines whether an incentive is offered.
        This incentive offer is based on analyzing the incentive levels for each neighboring zone.
        If a neighboring zone has a higher incentive level than the intended initial drop-off zone, an incentive is offered.
        If all neighboring zones have a lower incentive level, then no incentive is offered.
        Incentives are only offered for the neighboring cell with the highest incentive level.
        Instead of modeling a complex choice among multiple options, it is assumed that users are presented only with an incentive for the most attractive neighboring zone.
        The willingness of the user to take the incentive is modeled by a linear function (\texttt{user\_willingnes\_fn}).
        This linear function multiplies the offered incentive value [0, 1] by the willingness factor of 0.4, resulting in a linear willingness between 0 and 0.4.
        This simplified model is a deliberate methodological choice, consistent with state-of-the-art research in RL for rebalancing, which prioritizes the evaluation of control algorithms over high-fidelity behavioral prediction \cite{pan_deep_2019}.
        The linear form represents the simplest positive correlation between incentive and acceptance, while the 0.4 factor is a conservative parameterization.
        This willingness value then represents the probability that an individual user will accept the offered incentive.
        As a result, the function returns an updated list of drop-off events, including those shifted due to incentives, and the number of rebalanced vehicles.
    \item \textbf{Demand Simulation:}
        After all incentives are processed, the demand is simulated, resulting in updated vehicle counts per zone.
        The demand simulation works similarly to the \gls{rdc}'s environment.
        The demand is simulated sequentially, continuously updating the vehicle counts for the entire 60-minute step.
    \item \textbf{Observation and Reward Calculation:}
        Lastly, the reward is calculated based on the number of rebalanced vehicles, the satisfied demand, and the distribution of vehicles (see \autoref{sec:design-rl_framework-uic}).
        Then, a new state observation is created, and the method returns the complete experience tuple ($\mathcal{S}_t^{UIC}$, $\mathcal{A}_t^{UIC}$, $\mathcal{R}_t^{UIC}$, $\mathcal{S}_{t+1}^{UIC}$, $done^{UIC}$).
\end{enumerate}

\subsection{Training procedure}
The training of the \gls{uic} agent is orchestrated by the \texttt{OnPolicyAlgorithm.learn()} function, a core component of the \gls{ppo} implementation within the \gls{sb3} library \cite{raffin_stable-baselines3_2021}.
This procedure unfolds in two primary phases: experience collection and policy optimization.\\

Initially, during the experience collection phase, the agent interacts with the \texttt{EScooterUICEnv} environment to gather a batch of experiences.
This rollout is conducted over a number of timesteps defined by the \texttt{UIC\_n\_steps} hyperparameter.
At each timestep $t$, the agent observes the current state $S_t$, samples an action $\mathcal{A}_t$ based on its existing policy $\pi_{\theta_{old}}$, and executes this action.
The resultant experience tuple, denoted as ($\mathcal{S}_t, \mathcal{A}_t, \mathcal{R}_t, \mathcal{S}_{t+1}, \text{done}$), is systematically stored in a \texttt{RolloutBuffer} for subsequent processing.

Following the collection of experiences, the training process transitions to calculating the advantage estimates and returns necessary for the policy update.
The advantage $\hat{A}_t$ is computed for each timestep using \gls{gae}, which relies on the \gls{td-error}, $\delta_t$:

\begin{equation}
    \delta_t = \mathcal{R}_t + \gamma \cdot V_{\phi}(\mathcal{S}_{t+1}) \cdot (1 - \text{done}) - V_{\phi}(\mathcal{S}_t)
    \label{eq:uic_td_error_revised}
\end{equation}

Here, $\gamma$ represents the discount factor, a hyperparameter specified as \texttt{UIC\_gamma}.
The \gls{gae} advantage, $\hat{A}_t$, is then calculated as the discounted sum of these \glspl{td-error}, where the hyperparameter $\lambda_{\text{GAE}}$ (\texttt{UIC\_gae\_lambda}) modulates the trade-off between bias and variance \cite{schulman_high-dimensional_2018}:

\begin{equation}
    \hat{A}_t = \sum_{l=0}^{N-1} (\gamma \lambda_{\text{GAE}})^l \delta_{t+l}
    \label{eq:uic_gae_revised}
\end{equation}

The target value for the value function update, $G_t$, is subsequently derived from the advantage estimate and the current value function output, $V_{\phi}(\mathcal{S}_t)$, such that $G_t = \hat{A}_t + V_{\phi}(\mathcal{S}_t)$.\\

The core learning phase is encapsulated within the \gls{ppo}-specific \texttt{train()} method, which initiates an optimization loop \cite{raffin_stable-baselines3_2021}.
This loop iterates over the entire \texttt{RolloutBuffer} for a predefined number of epochs (\texttt{UIC\_n\_epochs}), shuffling the collected data and partitioning it into mini-batches of size \texttt{UIC\_batch\_size}.
For each mini-batch, the composite \gls{ppo} loss function, $L_{\text{total}}$, is evaluated.
This requires re-computing action log-probabilities, state-values, and the policy entropy with the current network parameters.
The total loss integrates three weighted components: the policy loss ($L^{\text{CLIP}}$), the value function loss ($L^{\text{VF}}$), and an entropy bonus ($S$) \cite{schulman_proximal_2017}:

\begin{equation}
    L_{\text{total}}(\theta, \phi) = L^{\text{CLIP}}(\theta) - c_{VF} \cdot L^{\text{VF}}(\phi) + c_{ent} \cdot S[\pi_{\theta}](\mathcal{S}_t)
    \label{eq:uic_total_loss_revised}
\end{equation}

The policy loss, or clipped surrogate objective, is governed by the \texttt{UIC\_clip\_range} hyperparameter and serves to stabilize policy updates.
The value function loss, scaled by the coefficient $c_{VF}$ (\texttt{UIC\_vf\_coeff}), is the mean squared error between the predicted state-values and the target returns $G_t$.
The entropy bonus, scaled by $c_{ent}$ (\texttt{UIC\_ent\_coeff}), encourages exploration by maximizing the policy's randomness.
The computed total loss is then backpropagated to determine the gradients.
An Adam optimizer \cite{kingma_adam_2017}, configured with the \texttt{UIC\_lr} learning rate, executes a single optimization step, updating the parameters ($\theta$) of both the policy and value networks.
Finally, a learning rate scheduler updates the learning rate according to a predefined linear schedule, completing the training cycle.

\section{Demand Forecasting Module}\label{sec:implementation_demand-forecasting-module}
A requirement for the effective operation of the \gls{rdc} and \glspl{uic} is an accurate forecast of future demand.
This is provided by the Demand Forecasting Module, the implementation of which is detailed in this section.
This Demand Forecasting Module is built upon the implementation by Li et al. \cite{li_improving_2023}, whose design was discussed in \autoref{sec:design-rl_framework-forecasting_module}.
For the scope of this thesis, the original hyperparameters from Li et al. were adopted under the assumption of reasonable transferability.
A complete hyperparameter optimization for the forecasting module was considered out of scope, but is noted as a potential area for future improvement.
This pragmatic decision was made to prioritize the project's core contribution, which is the design and evaluation of the HRL framework itself.
However, it is crucial to acknowledge that forecasting performance is critical for the agents' decision-making.
Non-optimal hyperparameters for the Stuttgart dataset could potentially impact the overall performance.

To prepare the demand data, the preprocessed trip data was cleaned by adding missing timesteps, filling null values with zero demand, and sorting community IDs.
The final data structure for the model input was a tensor of shape $(|\mathcal{Z}|, T_{input}, 1)$, where $T_{input}$ is the number of historical timesteps and the third dimension represents the scalar demand.
The full implementation of this process is detailed in the Jupyter Notebook \url{prepare_demand_forecasting_data.ipynb}.
A similarity matrix is required to define the kernels for each zone.
This similarity matrix is generated by calculating a similarity metric based on the \gls{dtw} distance between the demand of all zones, resulting in a matrix of size $(|\mathcal{Z}| \times |\mathcal{Z}|)$ \cite{li_improving_2023}.
Based on this distance parameter (\texttt{dtw\_dist}), the similarity is calculated (see \autoref{eq:irconvlstm-similarity}).
\gls{dtw} was chosen instead of \gls{pcc} due to its superior performance results \cite{li_improving_2023}.

\begin{equation}\label{eq:irconvlstm-similarity}
    \texttt{similarity} = \frac{1}{1 + \texttt{dtw\_dist}}
\end{equation}

A kernel size of seven has been chosen.
For each zone, its convolutional kernel was constructed using itself and the six zones with the highest demand similarity, as determined by the \gls{dtw}-based similarity matrix.

For training the agents and hyperparameter optimization, the forecasts have been pre-computed for the entire training set, which significantly improved computational performance and, therefore, resulted in faster training.
