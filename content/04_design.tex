\section{Data}
The data underlying this thesis was collected from a General Bike Sharing Specification (GBFS) feed that publishes the live locations of e-scooters from sharing services operating in the greater Stuttgart area \cite{noauthor_gebundelte_nodate}. These services utilize a free-floating approach, permitting users to pick-up and drop-off vehicles at any location within the operating area, without restriction to specific parking stations. The raw GBFS data undergoes a pre-processing stage to extract pick-up and drop-off events. These events were subsequently spatially and temporally aggregated using the H3 grid system \cite{noauthor_h3_2018} and hourly time bins for analysis, as well as for training and evaluating the proposed rebalancing framework.

\subsection{Preprocessing}
The raw GBFS data is preprocessed to extract pick-up and drop-off events.
Both GBFS feeds randomize vehicle identification for anonymization, making it impossible to extract information on the actual trips driven by the users.
Therefore, only pick-up and drop-off events were extracted, which reduced the usability of this data.
To extract those events, neighbouring snapshots are compared.
Vehicle Identifiers that disappear or reappear in a snapshot compared to the previous snapshot reveal the occurrence of pick-ups and drop-offs.
After those events were derived from the datasets, 200,000 pick-up and drop-off events for the service bolt and 590.000 pick-up and drop-off events for the service provider Voi were counted.
After that, outliers have been removed from the dataset. Therefore, a boundary for the city of Stuttgart has been defined to filter the pick-up and drop-off events that occurred only inside this boundary. This has been done to focus on a smaller region instead of outer city regions like Leinfelden.

\section{Problem Definition}
This section formalizes the fleet rebalancing problem of a free-floating e-scooter sharing system.
The problem definition considers both operator-based rebalancing, performed by service workers, and user-based rebalancing, relocations performed by users guided by incentives.
It considers only the rebalancing in the form of recommended incentives per zone and net vehicle counts per community.
The routing problem of service vehicles performing operator-based rebalancing is not addressed in this work. 
Several works have been published on the routing problem for operator-based rebalancing, so there is no need to account for that.   

\subsection{Scope and Notation}
The service area is partitioned into a set of zones($\mathcal{Z}$).
The time is discretized into hourly intervals ($\mathcal{T}$) in which all pick-up, drop-off, and rebalancing events occur.
The fleet size ($\mathcal{N}$) remains constant over all timesteps $t$.
\autoref{tab:problem_notation} summarizes all notation used for the problem formulation, including all sets, parameters, and decision variables.

\subsection{State Variables}
The formulated rebalancing problem distinguishes between forecasted demand ($\hat P_z(t)$, $\hat D_z(t)$), which forms part of the RL agent's state at each decision, and actual demand ($P_z(t)$, $P_z(t)$), which drives the simulation used for training and evaluation.
All performance metrics, such as the satisfied demand, are computed against the actual demand values.

\subsection{Rebalancing actions}
The framework considers both operator-based and user-based rebalancing, which is why the problem definition includes decision variables for both approaches.
Those decision variables are described in the following paragraphs.

\subsubsection{Operator-based rebalancing}
Operator-based rebalancing is performed to relocate e-scooters between communities.
Service workers move e-scooters between communities, which is represented by $X_{c\rightarrow c'}(t)$, describing the number of vehicles relocated from community $c$ to community $c'$ at time step $t$.
These relocations incur a cost defined by $c^{op}_{c\to c'}$ per relocated e-scooter.

\subsubsection{User-based rebalancing}
User-based rebalancing is performed between zones inside a community.
Users are incentivized to drop-off vehicles at a different zone than they originally intended to.
The decision variable $Y_{z\rightarrow z'}(t)$ describes such a relocation done by a user from zone $z$ to $z'$.
User-based relocations also incur a cost, which is defined by $c^{user}_{z \to z'}$.
\autoref{eq:rb-user-constraint-spatial} constraints user-based rebalancing to take part only inside a community.
Additionally, user-based rebalancing operations are restricted to the neighboring zones.
\begin{equation}\label{eq:rb-user-constraint-spatial}
Y_{z\to z'}(t) \;=\; 0
\quad
\forall\,t\in\mathcal{T},\;\forall\,z,z'\in\mathcal{Z}
\text{ such that }
\bigl(A_{z,z'}=0\bigr)\;\lor\;\bigl(\kappa(z)\neq\kappa(z')\bigr).
\end{equation}

\subsection{System Dynamics}
 

\autoref{eq:rp-fleet-conservation} formulates the global fleet conservation.
This constraint ensures that the total number of e-scooters across all zones remains constant at every time period $t$.

\begin{equation}\label{eq:rp-fleet-conservation}
    \sum_{z \in \mathcal{Z}}n_z(t) = \mathcal{N}, \forall t\in \mathcal{T}
\end{equation}


\autoref{eq:pb-demand-flow} illustrates the demand flow of a single zone, defining the number of e-scooters at time $t+1$ by accounting for demand and relocation operations.
This demand flow enables a step-wise calculation of available scooters at every timestep $t$.
The demand flow considers both pick-up demand $P_z(t)$ and drop-off demand $D_z(t)$.
Additionally, rebalancing operations are considered by the decision variables $X_{c\rightarrow c'}(t)$ and $Y_{z\rightarrow z'}(t)$.
\begin{equation}\label{eq:pb-demand-flow}
    \begin{aligned}
        n_z(t+1) =\;&\ n_z(t) - P_z(t) + D_z(t) \\
        &+  \sum_{c' \in \mathcal{C}} \frac{1}{|\mathcal{Z}_{\kappa(z)}|} X_{c'\rightarrow \kappa(z)}(t) - \sum_{c' \in \mathcal{C}} \frac{1}{|\mathcal{Z}_{\kappa(z)}|} X_{\kappa(z)\rightarrow c'}(t) \\
        &+ \sum_{z' \in \mathcal{Z}} Y_{z'\rightarrow z}(t) - \sum_{z' \in \mathcal{Z}} Y_{z\rightarrow z'}(t)
    \end{aligned}
\end{equation}

\subsection{Objective}

\autoref{eq:problem-definition-satisfied-demand} defines the satisfied demand, which is the demand that the service could satisfy with enough vehicles to serve the wanted pick-up demand.
This satisfied demand is used to calculate the ratio of satisfied demand, which defines the percentage of the asked demand to the amount of demand that could be satisfied by the system (see \autoref{eq:problem-definition-satisfied-demand-ratio}).

\begin{subequations}
    \begin{equation}\label{eq:problem-definition-satisfied-demand}
        S_z(t) = \min\{P_z(t), n_z(t) + D_z(t)\}
    \end{equation}
    \begin{equation}\label{eq:problem-definition-satisfied-demand-ratio}
        r^S_z(t) = 
        \begin{cases}
            \frac{S_z(t)}{P_z(t)}, & P_z(t) > 0,\\[8pt]
            0, & P_z(t) = 0.
        \end{cases}
    \end{equation}
\end{subequations}

\autoref{eq:pb-objective} defines the objective of the rebalancing problem.
It combines the costs from rebalancing operations performed by user- and operator-based rebalancing, and includes minimizing the amount of unmet demand.
The objective is calculated over the whole simulation horizon, e.g., over all time steps $t$.

\begin{equation}\label{eq:pb-objective-gini}
G(t) \;=\; 
\frac{1}{2 \, \mathcal{N}_z^2 \, \bar{n}(t)} 
\sum_{i=1}^{\mathcal{N}_z} \sum_{j=1}^{\mathcal{N}_z} 
\bigl\lvert n_i(t) - n_j(t)\bigr\rvert,
\qquad
\bar{n}(t) \;=\; \frac{1}{\mathcal{N}_z} \sum_{z=1}^{\mathcal{N}_z} n_z(t).
\end{equation}

\section{Framework}
A hierarchical reinforcement learning (HRL) framework is proposed to address the e-scooter rebalancing challenge.
The HRL framework comprises two tiers.
The first tier, the Regional Distribution Coordinator, manages inter-community imbalances by determining the target number of vehicles for each defined community.
These imbalances are addressed through operator-based rebalancing, where service workers collect vehicles from communities with an oversupply and redistribute them to communities experiencing an undersupply.
The second tier consists of multiple User-Incentive Coordinators, each assigned to a specific community, to manage intra-community imbalances (i.e., imbalances within a single community).
These coordinators aim to promote an equal distribution of vehicles by offering incentives to users for returning vehicles to designated undersupplied zones at the end of their trips.

A dedicated demand forecasting module supports both the Regional Distribution Coordinator and the User-Incentive Coordinators.
This module provides demand predictions as part of the state for both tiers, enhancing the effectiveness of rebalancing operations by allowing the system to anticipate future demand.
This modular design ensures that both tiers optimize rebalancing decisions based on consistent future outlooks and separates the demand forecasting task from the RL agents.
Additionally, the current vehicle distribution across all zones is provided as input to both agents. \autoref{fig:hrl-architecture} illustrates the architecture of this two-tier HRL framework.

\subsection{Regional Distribution Coordinator}\label{sec:design-rl_framework-rdc}
The Regional Distribution Coordinator (RDC) constitutes the first tier of the proposed two-tier Hierarchical Reinforcement Learning (HRL) framework.
Its primary purpose is to address inter-community imbalances by determining the target number of e-scooters for each defined community, facilitating operator-based rebalancing.

\subsubsection{Markov Decision Process (MDP) formulation}
The RDC is modeled as a Markov Decision Process (MDP), defined by a tuple $(\mathcal{S, A, P, R})$, where $\mathcal{S}$ is the state space, $\mathcal{A}$ is the action space, $\mathcal{P}$ is the state transition probability function, and $\mathcal{R}$ is the reward function \cite{sutton_reinforcement_2018}.
Due to the complex dynamics of shared e-scooter systems, the transition probability function $\mathcal{P}$ is learned implicitly by the agent through interaction with the simulation environment.
The MDP is assumed to be stationary, meaning that the transition probabilities depend only on the current state and the selected action and not on the specific timestep $t$.
This MDP formulation is illustrated in \autoref{fig:rdc_mdp}.

\paragraph{State ($\mathcal{S}^{RDC}_t$)}
The state representation for the RDC at timestep $t$, denoted $\mathcal{S}^{RDC}_t$, is a triple containing community-level vehicle counts ($n_c(t)$), pick-up demand forecast ($\hat{P}_c(t)$), and drop-off demand forecast ($\hat{D}_c(t)$).
These demand forecasts are generated by a separate demand forecasting module, as detailed in \autoref{sec:design-rl_framework-forecasting_module}.


\begin{itemize}
    \item \textbf{Community-level Vehicle Vounts ($n_c(t)$):}
    The current number of available e-scooters in the community $c$ at timestep $t$.
    \item \textbf{Pick-up Demand Forecast ($\hat{P}_c(t)$):}
    The forecasted number of e-scooter trips expected to originate from community $c$ during the next time interval ($t \to t+1$).
    \item \textbf{Drop-off Demand Forecast ($\hat{D}_c(t)$):}
    The forecasted number of e-scooter trips expected to end in community $c$ during the next time interval ($t \to t+1$).
\end{itemize}

\paragraph{Action ($\mathcal{A}^{RDC}_t$)}
The RDC's action at timestep $t$, $\mathcal{A}^{RDC}_t$, is a vector $[a_1(t), a_2(t), ..., a_{N_{\mathcal{C}}}(t)]$, where $N_{\mathcal{C}}$ is the total number of communities.
Each component $a_c(t)$ specifies the rebalancing quota for community $c$, representing the target number of e-scooters to be added to (positive value) or removed from (negative value) that community.
The action space for each component $a_c(t)$ is discrete and identical across communities. For this study, the action space for each $a_c(t)$ is defined as $\{-15, -10, -5, 0, 5, 10, 15\}$, providing seven distinct rebalancing options per community.

The RDC aims to select actions that balance supply across all communities. While ideal actions might inherently preserve the overall fleet size, the RDC can select a set of actions that, if summed, would change the total number of vehicles.
Such scenarios are managed by the operator-rebalancing algorithms within the simulation environment, which execute the rebalancing operations. If necessary, the simulator scales the raw action values (quotas) proportionally to ensure that the total fleet size remains constant, thereby preserving the RDC's intended rebalancing direction while sticking to operational constraints.

\paragraph{Reward ($\mathcal{R}^{RDC}_t$)}
With state and action now defined, we next specify how the RDC quantifies its performance via the reward function.
The reward function of the RDC is designed to guide the agent towards effective rebalancing strategies.
It is calculated after the execution of the rebalancing action $\mathcal{A}^{RDC}_t$ and the simulation of user demand flow for the current timestep. The total reward $\mathcal{R}^{RDC}_t$ is a weighted sum of three components.
These components reflect the primary operational goals: maximizing service utility, minimizing operational costs, and ensuring equitable vehicle availability.

\begin{equation}\label{eq:rdc-reward}
    \mathcal{R}^{RDC}_t = \frac{w_{demand}^{RDC} \cdot R_{demand}^{RDC}(t) + w_{cost}^{RDC} \cdot R_{cost}^{RDC}(t) + w_{gini}^{RDC} \cdot R_{gini}^{RDC}(t)}{w_{demand}^{RDC}+ w_{cost}^{RDC}+ w_{gini}^{RDC}}
\end{equation}
where $w_{demand}^{RDC}$, $w_{cost}^{RDC}$, and $w_{gini}^{RDC}$ are non-negative weighting factors. Each component reward ($R_{demand}^{RDC}(t)$, $R_{cost}^{RDC}(t)$, $R_{gini}^{RDC}(t)$) is individually normalized to a range of [0, 1].
The specific values for the weighting factors and the methodology for their selection are detailed in \autoref{sec:implementation_rdc}.

\begin{subequations}
    \begin{equation}\label{eq:rdc-reward-demand}
        R_{demand}^{RDC}(t) = \frac{\sum_{z \in \mathcal{Z}}S_z(t)}{\sum_{z \in \mathcal{Z}}D_z(t)}
    \end{equation}
    \begin{equation}\label{eq:rdc-reward-cost}
        R_{cost}^{RDC}(t) = 1 -  \left(\frac{\sum_{c, c' \in \mathcal{C}}X_{c \to c'}(t)}{\mathcal{N}}\right)^2
    \end{equation}
    \begin{equation}\label{eq:rdc-reward-gini}
        R_{gini}^{RDC}(t) = 1 - G(t)
    \end{equation}
\end{subequations}

\begin{enumerate}
    \item \textbf{Satisfied Demand Ratio ($R_{demand}^{RDC}(t)$):}
    This core component quantifies the effectiveness of rebalancing actions by measuring the proportion of satisfied trip requests.
    It is calculated by summing, over all communities, the number of trip requests the system successfully served, and dividing this sum by the total number of trip requests received during the corresponding time interval (see \autoref{eq:rdc-reward-demand}).
    This normalization yields a stable reward signal between 0 and 1.
    \item \textbf{Rebalancing Cost ($R_{cost}^{RDC}(t)$):}
    To penalize excessive rebalancing, this component incorporates the cost of rebalancing operations.
    It is calculated by summing the total number of vehicles rebalanced (picked up or dropped off by operators) in the current timestep.
    This sum is then divided by the total fleet size to obtain a normalized rebalancing ratio.
    This ratio is squared and then transformed to ensure that a higher number of rebalancing operations results in a lower value for this reward component, and vice versa (see \autoref{eq:rdc-reward-cost}).
    \item \textbf{Imbalance Penalty ($R_{gini}^{RDC}(t)$):}
    This component encourages an equitable distribution of the e-scooter fleet across communities.
    It is derived from the Gini coefficient, a measure of statistical dispersion, calculated over the vehicle counts in all communities.
    The Gini coefficient itself ranges from 0 (perfect equality) to 1 (perfect inequality).
    This value is then transformed to align with the reward structure (see \autoref{eq:rdc-reward-gini}), resulting in a normalized component that is high when the e-scooter fleet is evenly distributed and low for an imbalanced distribution \cite{cederle_fairness-oriented_2025}.
\end{enumerate}

\subsubsection{Algorithm and Architecture}
Given this MDP formulation, we implement a Deep Q‐Network to approximate the optimal policy.
A multi-head Deep Q-Network (DQN) is employed for the Regional Distribution Coordinator. DQN is a value-based reinforcement learning algorithm that utilizes deep neural networks to approximate the optimal action-value function, $Q*(s, a)$ \cite{mnih_human-level_2015}.
The multi-head variant \cite{li_personalized_2024} is particularly suitable for the RDC as it needs to output multiple, correlated rebalancing actions (one for each community) simultaneously.
Alternative architectures, such as a single network predicting a composite action vector or Q-values for all joint actions, were considered but deemed impractical due to the combinatorial explosion of the action space.

\begin{figure}[!htb]
    \centering
    \includegraphics[width=\linewidth]{content/figures/mh-dqn-architecture.pdf}
    \caption{Multi‐head DQN architecture for the RDC, with shared layers and per‐community heads.}
    \label{fig:mdh-architecture}
\end{figure}

The core architecture of the multi-head DQN comprises three main types of layers:
\begin{enumerate}
    \item \textbf{Input Layer:}
    Processes the state representation ($\mathcal{S}^{RDC}_t$), involving flattening and normalization of the input features (community-level vehicle counts and demand forecasts).
    \item \textbf{Shared Layers:}
    These layers receive input from the Input Layer and learn to extract features representing system-wide metrics and relationships relevant to all communities.
    \item \textbf{Head-Specific Layers:}
    A separate set of neural network layers (a \textit{head}) is dedicated to each community.
    Each head takes the processed features from the Shared Layers and outputs the Q-values for each discrete rebalancing action $a_c(t)$ applicable to its corresponding community.
    The action selected for a community is the one with the highest Q-value from its respective head.
\end{enumerate}
Each head thus determines the optimal rebalancing quota for its assigned community based on the global system state and learned rebalancing policies.

\paragraph{Prioritized Experience Replay (PER)}
The multi-head DQN utilizes Prioritized Experience Replay (PER) \cite{schaul_prioritized_2016} to enhance learning efficiency.
Standard DQN stores transitions $(s_t, a_t, r_t, s_{t+1})$ in a replay buffer and samples them uniformly for training.
With PER, transitions are sampled non-uniformly based on their significance, measured by the magnitude of the Temporal Difference (TD) error.
Prioritizing transitions with high TD-error, which are transitions where the agent learns the most, accelerates the learning process and improves sample efficiency.
However, this prioritization introduces a bias by modifying the distribution of training data.

This bias is mitigated using Importance Sampling (IS) weights, which adjust the contribution of each sampled transition during the learning updates.
These IS-weights are modulated by an exponent $\beta_{PER}$, which is annealed from an initial value towards one over the course of training (see \autoref{sec:implementation_rdc} for the exact annealing schedule).
This annealing schedule for $\beta_{PER}$ helps to balance the benefits of prioritized sampling with the need for unbiased updates, contributing to stable and efficient learning \cite{schaul_prioritized_2016}.

\paragraph{Exploration Strategy}
To ensure sufficient exploration of the state-action space, the RDC employs an $\epsilon$-greedy strategy.
In $\epsilon$-greedy, $\epsilon$ represents the probability of choosing a random action rather than the action considered optimal by the current policy.
To encourage broad exploration initially and gradually shift towards exploitation, $\epsilon$ is annealed over the course of the training process (see \autoref{sec:implementation_rdc} for the exact $\epsilon$ schedule).
It starts at a high value at the beginning of training and is reduced to a small, non-zero value by the end of training.
Maintaining a small $\epsilon$ ensures persistent exploration, which can be beneficial in complex and dynamic environments \cite{zhang_why_2020}.

\paragraph{Training stabilization techniques}
Several techniques are incorporated to stabilize the training of the DQN.
A separate target network, with an identical architecture to the policy network (the network being actively trained), is used to provide stable targets for updating the Q-value.
The target network's weights are not updated through backpropagation directly.
Instead, they are periodically updated using soft updates, where its weights ($\theta_{target}$) become a weighted average of its own current weights and the policy network's weights ($\theta_{policy}$): $\theta_{target} \leftarrow \tau \theta_{policy} + (1-\tau)\theta_{target}$, for a small $\tau$ (see \autoref{sec:implementation_rdc} for hyperparameter values).
This combination of a separate target network and soft updates contributes to stabilizing the training process.

To address the bias in Q-learning, where Q-values are frequently overestimated, a Double DQN (DDQN) algorithm \cite{hasselt_deep_2016} is integrated.
DDQN decouples action selection from action evaluation for the next state's value estimation.
Specifically, the policy network determines the optimal action for the next state, and the target network then evaluates the Q-value of this selected action.

Furthermore, to enhance performance and maintain training stability, a learning rate schedule is employed.
The learning rate starts higher to accelerate initial learning and is gradually decreased to facilitate fine-tuning and convergence to more optimal solutions.
Additionally, gradient clipping is applied during backpropagation to prevent exploding gradients, a common issue that can destabilize the training of deep neural networks \cite{sutton_reinforcement_2018}.

\subsection{User-Incentive Coordinator}\label{sec:design-rl_framework-uic}
Implemented with the PPO algorithm due to the large action space. Intra-community imbalances.


\subsubsection{Markov Decision Process (MDP) formulation}
\paragraph{State ($\mathcal{S}^{UIC}_t$)}
The state representation of the UIC is similarly defined as the state space of the RDC, but scoped to the specific community to which the UIC is designated.
The state representation consists of three components:

\begin{itemize}
    \item \textbf{Pick-up Demand Forecast ($\hat{P}_z(t)$):}
    The forecasted number of e-scooters that are expected to originate at zone $z$ during the next time interval ($t \to t+1$). Where the community $\kappa(z)$ is the designated community of the UIC.
    \item \textbf{Drop-off demand forecast ($\hat{D}_z(t)$):}
    The forecasted number of e-scooters that are expected to be dropped off by users, e.g., the number of trips that are ending at zone $z$ in the next time interval ($t \to t+1$). Where the community $\kappa(z)$ is the designated community of the UIC.
    \item \textbf{Zone-level vehicle counts ($n_z(t)$):}
    The current number of available e-scooters in the zone $z$ at the current time step $t$. Where the community $\kappa(z)$ is the designated community of the UIC.
\end{itemize}

\paragraph{Action ($\mathcal{A}^{UIC}_t$)}
The UIC's action space is continuous.
The UIC's actions represent continuous incentive levels between $[0,1]$ for each zone $z$ that is part of the designated community $c$.
This results in the following definition of the Actions:

\begin{equation}
    \mathcal{A}^{UIC}_t = [a_1^{UIC}(t), ..., a_{N_z^c}^{UIC}(t)]
\end{equation}

$N_z^c$ defines the number of zones in community $c$.



\paragraph{Reward ($\mathcal{R}^{UIC}_t$)}

The reward function $\mathcal{R}^{UIC}_t$ of the UIC considers three components, each weighted by a non-negative weighting factor.

\begin{equation}\label{eq:uic-reward}
    \mathcal{R}^{UIC}_t = w_{demand}^{UIC} \cdot R_{demand}^{UIC}(t) + w_{cost}^{UIC} \cdot R_{cost}(t) + w_{gini}^{UIC} \cdot R_{gini}^{UIC}(t)
\end{equation}

\subsubsection{Algorithm and Architecture}
A Proximal Policy Optimization (PPO)\cite{schulman_proximal_2017} algorithm is implemented for the User-Incentive Coordinators.
This algorithm allocates incentives to specific zones within a community to encourage users to drop-off vehicles in these targeted zones rather than their originally intended destinations.
The implementation leverages the Stable Baselines3 library, which provides an robust and stable implementation of the PPO algorithm \cite{raffin_stable-baselines3_2021}.

\begin{itemize}
    \item \textbf{State}
    \begin{itemize}
        \item Local demand forecast ($\hat{P_z}(t)$, $\hat{D_z}(t)$ $\forall z \in \mathcal{Z}_c$), local vehicle counts per grid cell($n_z(t)$ $\forall z \in \mathcal{Z}_c$)
    \end{itemize}
    \item \textbf{Action}
    \begin{itemize}
        \item Assigns price incentives for each grid cell for dropping off at these grid cells
    \end{itemize}
    \item \textbf{Reward}
    \begin{itemize}
        \item Satisfied demand ($s_z(t)$ $\forall z \in \mathcal{Z}_c$), negative incentive costs
    \end{itemize}
\end{itemize}

\subsection{Demand Forecasting Module}\label{sec:design-rl_framework-forecasting_module}
The successful operation of the rebalancing algorithm is fundamentally dependent on its ability to anticipate future demand.
Therefore, a dedicated Demand Forecasting Module that accurately predicts future demand is critical.
This implementation of an independent Demand Forecasting Module takes out the forecasting problem from the rebalancing algorithms themselves, which ideally should increase the overall efficiency of the framework since the problems are separated and can be targeted directly.

To effectively support the RL rebalancing agents mentioned earlier, the Demand Forecasting Module must predict the number of pick-ups and drop-offs for each zone on an hourly basis.
The underlying model of this Demand Forecasting Module is trained using historical demand data extracted from the raw GBFS data.
To predict both pick-up and drop-off demand two separate models are trained for each of them.
The output of both demand models should be the expected number of pick-up / drop-off events hourly per zone.

The model selected for the Demand Forecasting Module is the Irregular Convolutional Long Short-Term Memory (IrConvLSTM) network by Li et al. \cite{li_improving_2023}.
The selection of the IrConvLSTM model as part of this rebalancing framework is based on its empirically validated performance and its publicly available open-source implementation.
Due to the short time scope of this master's thesis, the focus of this work is on the rebalancing framework itself, which is why an open-source implementation for the demand forecasting module is preferred.
Li et al. \cite{li_improving_2023} demonstrated the high performance of the IrConvLSTM model in an empirical study comparing the model extensively against several other forecasting modules.
These included traditional time-series models like ARIMA, standalone deep learning models like LSTM and GRU, and hybrid models like the standard CNN+LSTM model.
The evaluation has been done over five different cities with bike-sharing data.
The IrCOnvLSTM using Dynamic Time Warping yielded the best performances, which is why this variant is used for this work.
IRConvLSTM consistently outperformed the other models across all benchmarks. including Root Mean Squared Error (RMSE), Mean Absolute Error (MAE), and Mean Absolute Percentage Error (MAPE).

The implementation details of this Demand Forecasting Module and the IrConvLSTM are described in \autoref{sec:implementation_demand-forecasting-module}.
